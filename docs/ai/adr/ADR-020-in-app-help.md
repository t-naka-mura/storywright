# ADR-020: アプリ内ヘルプの導入

- 日付: 2026-04-04
- ステータス: 提案中

## Context (背景)

Storywright は「非エンジニアでもインストールするだけで使える E2E テストツール」を目指している。しかし現状、アプリ内に操作ガイドやヘルプが存在せず、初めて触るユーザーが以下の点で迷う可能性がある:

- REC ボタンを押した後に何をすればいいのか
- Assert モードの使い方
- `{{ENV.API_KEY}}` のような環境変数プレースホルダの書き方
- Story の export / import の手順
- 繰り返し実行や keepSession の意味

エンジニアであればUIから推測できるが、非エンジニアのQA担当やディレクターが使う場合、最初の一歩でつまずく可能性がある。

一方で、アプリの機能はまだ発展途上であり、ヘルプコンテンツのメンテナンスコストも考慮する必要がある。

## Options (選択肢)

### A. アプリ内ヘルプ view を追加する

Settings と同様に独立ウィンドウまたはパネルでヘルプを表示。主要な操作フローをステップバイステップで説明する静的コンテンツ。

**メリット**
- オフラインでも参照できる
- アプリの世界観（Warm Functional）に統一したデザインで提供できる
- 初回起動時に自動表示する等のオンボーディング導線を作れる

**デメリット**
- 機能追加のたびにヘルプコンテンツの更新が必要
- 実装・デザインコストがかかる
- 機能が安定するまでコンテンツが頻繁に陳腐化する

### B. コンテキストヒント（ツールチップ / インラインヘルプ）

ヘルプ画面を設けず、各 UI 要素にツールチップやインラインの説明テキストを追加。例: REC ボタンの横に「ブラウザ操作を録画します」、環境変数入力欄に `{{ENV.KEY}}` の使い方の説明。

**メリット**
- 操作の文脈で即座に情報が得られる
- 独立したヘルプ画面のメンテナンスが不要
- 段階的に追加できる（一気に全部作る必要がない）

**デメリット**
- 体系的な操作フローの説明には向かない
- UI がごちゃつくリスク
- 「全体の流れ」を把握するには不十分

### C. 外部ドキュメントへのリンク

メニューバーやヘルプボタンから Web 上のドキュメント（GitHub Pages 等）を開く。

**メリット**
- アプリ本体の実装コストがほぼゼロ
- ドキュメントの更新がアプリのリリースと独立
- 検索可能

**デメリット**
- オフライン環境で参照できない
- 外部サイトの構築・メンテナンスが別途必要
- アプリ内の体験が途切れる

### D. 今は作らない

機能がまだ頻繁に変わるフェーズなので、ヘルプコンテンツの陳腐化リスクが高い。ユーザーからのフィードバックが集まってから必要性を再判定する。

**メリット**
- 開発リソースをコア機能に集中できる
- 陳腐化したヘルプが逆にユーザーを混乱させるリスクを回避

**デメリット**
- 非エンジニアユーザーの最初の一歩を支援できない
- 「使い方がわからない」で離脱するリスク

## Decision (決定)

未決定。以下の論点で判断したい:

1. **タイミング**: 機能がまだ変わるフェーズで作るべきか？ → D の「今は作らない」も有力
2. **最小形**: 作るなら B（コンテキストヒント）を先行し、必要に応じて A に発展させるのが現実的か
3. **対象ユーザー**: 実際に非エンジニアに使ってもらう予定が近いか？ → 予定が遠いなら D の優先度が上がる

組み合わせとして **D → B → A** の段階的アプローチ（今は作らず、ユーザーが増えたらコンテキストヒントから始め、必要ならヘルプ view へ）も選択肢。

## Consequences (影響)

### A を採用した場合
- Settings と同様の独立ウィンドウの仕組みを流用できる
- `docs/ai/` のコンテンツをベースに初期ヘルプを生成できる
- デザイン原則（Warm Functional）に沿ったヘルプ画面のデザインが必要

### B を採用した場合
- 既存コンポーネントに `title` 属性やツールチップを追加する軽量な変更
- UI の情報密度が上がるため、デザインの慎重なバランス調整が必要

### D を採用した場合
- 影響なし。次のマイルストーン（非エンジニアへの提供開始等）で再検討
