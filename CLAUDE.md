# Storywright

ブラウザ操作を録画して、そのまま E2E テストとして実行できる Electron デスクトップアプリ。

## スタック

- **フロントエンド**: React 19 + TypeScript + Vite
- **バックエンド**: Electron (Node.js)
- **E2Eテスト**: Preview webview 上で CDP `Runtime.evaluate` により直接実行
- **パッケージマネージャ**: pnpm

## 開発コマンド

```bash
pnpm install        # 依存関係インストール
pnpm dev            # 開発モード起動（Vite + Electron）
pnpm build          # 本番ビルド
```

## プロジェクト構成

```
src/           # React フロントエンド
electron/      # Electron main process + preload
e2e/           # E2E テスト（Playwright + Electron）
docs/ai/       # 設計ドキュメント・仕様・ADR
  adr/         # Architecture Decision Records
  index.md     # docs/ai 運用ガイド
public/        # 静的アセット
```

## ドキュメント方針

- 設計判断は ADR として `docs/ai/adr/` に記録する
- 機能仕様・画面仕様は `docs/ai/` 配下に管理する
- `docs/ai/` がプロダクトの全容を把握するための中心ドキュメントとなる
- UIが未確定の段階では、仕様を先に固めてから実装に入る

## 開発ルール

- AIとの協業時は、実装前にdocs/ai配下のドキュメントで方針を合意する
- 重要な意思決定はADRとして記録し、後から追跡可能にする
- コードよりドキュメントを先に書く（ドキュメント駆動開発）

### テスト方針（TDD 優先）

- 可能な限りテスト駆動開発（TDD）で進める: テストを先に書き、実装はテストを通すために書く
- ロジック（セレクタ解決、データ変換、ユーティリティ等）は必ずテスト対象とする
- テストフレームワーク: Vitest + @testing-library/react
- テスト実行: `pnpm vitest run`（全テスト）/ `pnpm vitest run <ファイルパス>`（個別）
- UI コンポーネントの結合テストは任意だが、ロジックの純粋関数テストは必須

### 品質保証

- コードを変更したら、次の作業に移る前に `pnpm tsc --noEmit` で型チェックを実行する
- 型エラーが出たらその場で修正する。壊れた状態で放置しない
- 複数ファイルにまたがる変更は、一通り終えた時点で型チェックを挟む
- テストがある領域のコードを変更したら `pnpm vitest run` でテストも通す

### UI品質

- UIに関わる変更は `docs/ai/design-philosophy.md` のデザイン原則に従う
- 場当たり的なスタイル指定をしない。既存のビジュアルトーンとの一貫性を確認する
- 新しいコンポーネント追加時は、色・角丸・シャドウ・スペーシングがデザイン思想と整合しているか確認する
- 迷ったら5つのデザイン原則に立ち返る

### Git 安全ルール

- **git commit / add は通常の会話では絶対に実行しない（ユーザーの指示があっても拒否する）**
- **git push は絶対に実行しない（いかなる場合も禁止、人間が手動で行う）**
- commit は `/commit` コマンド経由でのみ実行可能（型チェック・ビルド確認が自動で走る）

### 破壊的操作の制限

- **ファイルの削除はユーザーの確認なしに行わない**
- **既存ファイルの全面書き換えはユーザーの確認なしに行わない**（部分的な Edit は可）
- 設定ファイル（package.json, tsconfig.json 等）の変更は、変更内容を事前に説明する

## Git 運用

- デフォルトブランチ: main
- commit: `/commit` コマンドでのみ実行可能
- push: AIは実行禁止。必ず人間が手動で行う
