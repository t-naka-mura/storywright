# ADR-011: ストーリーの繰り返し実行（リピート実行）

- 日付: 2026-04-03
- ステータス: 完了

## Context (背景)

### 現状の課題

現在のテスト実行は「1ストーリー = 1回実行」に固定されている。しかし、以下のようなユースケースに対応できない:

1. **データ量産**: 同じ購入フローを繰り返し実行して、テスト用の注文データを大量に作りたい
2. **安定性検証**: 同じシナリオを複数回実行して、フレーク（不安定）なステップを検出したい
3. **負荷確認**: 同一操作の反復で、対象サイトの挙動を確認したい

### 将来の拡張（Phase B）

繰り返し実行の枠組みが整えば、次のステップとして「実行ごとに値を差し替えるデータドリブン実行」への拡張が自然にできる。本 ADR は Phase A（シンプルループ）をスコープとし、Phase B（データドリブン）への拡張パスを意識した設計にする。

## Options (選択肢)

### A. runStory の外側にループを追加（シンプルラッパー）

- `run-story` IPC ハンドラの外側に繰り返しループを追加
- 毎回ブラウザを起動→ステップ実行→ブラウザ終了を繰り返す
- 各イテレーションの結果を個別に返却

**メリット**: 実装がシンプル、各回が完全に独立しているので副作用がない
**デメリット**: 毎回ブラウザ起動のオーバーヘッドがある

### B. ブラウザを使い回してページだけリセット

- ブラウザインスタンスは1つ、イテレーションごとに新しいページ（タブ）を作成
- Cookie やストレージはコンテキストごとにリセット

**メリット**: 起動コスト削減で高速
**デメリット**: ブラウザ状態のリーク リスクがある、デバッグが複雑になる

### C. 繰り返し回数を Story モデルに持たせる

- `Story` 型に `repeatCount` フィールドを追加し、永続化する
- 実行時にこの値を参照

**メリット**: ストーリーごとに繰り返し設定を保存できる
**デメリット**: Story のデータモデルを変更する必要がある、テスト定義と実行設定の責務が混ざる

## Decision (決定)

**A + C のハイブリッド: ブラウザは毎回起動し直し、繰り返し回数は実行時パラメータとして渡す。**

Story モデル自体は変更せず、実行時のオプションとして `repeatCount` を渡す設計にする。理由:

- Story はテストの「定義」、繰り返し回数は「実行設定」— 責務を分離する
- Phase B でデータソースを差し込む際も、実行時パラメータとして渡す形が自然
- ブラウザ毎回起動は遅いが、データ量産ユースケースでは正確性（状態リークなし）が優先

### 変更内容

#### 1. IPC インターフェース

```typescript
// 新しい IPC チャンネル
"run-story-repeat": (storyJson: string, repeatCount: number) => Promise<RepeatResult>

// または既存の run-story を拡張
"run-story": (storyJson: string, options?: { repeatCount?: number }) => Promise<StoryResult | RepeatResult>
```

#### 2. 型定義の追加（src/types.ts）

```typescript
// 1回分の実行結果（既存の StoryResult をそのまま使う）

// 繰り返し実行の全体結果
export type RepeatResult = {
  storyId: string;
  totalIterations: number;
  completedIterations: number;
  passedIterations: number;
  failedIterations: number;
  iterations: StoryResult[];  // 各回の結果
};
```

#### 3. メインプロセスの実行ロジック（electron/main.ts）

```
for i in 1..repeatCount:
  browser = launch()
  結果 = runStory(browser, story)
  iterations.push(結果)
  browser.close()
  進捗を renderer に通知（"repeat-progress" イベント）
  if ユーザーが中断要求 → break
```

- 各イテレーションでブラウザを新規起動（状態を完全リセット）
- 進捗はイテレーション単位で renderer にリアルタイム通知
- 中断機能を用意（長時間実行への対応）

#### 4. UI の変更（DetailPanel）

**設計方針**: 主操作（1回実行）のワンクリック性を守り、繰り返し実行は誤操作を防ぐ導線にする。

**スプリットボタン方式**:

```
┌──────────┬───┐
│  ▶ Run   │ ▼ │   ← 左クリック: 即1回実行（今と同じ）
└──────────┴───┘
               │
               ▼  ドロップダウン展開
         ┌─────────────────────┐
         │ 🔁 繰り返し実行...  │
         └─────────────────────┘
                    │
                    ▼  クリックで回数指定UIを表示
         ┌─────────────────────┐
         │ 繰り返し回数:       │
         │ [    10    ] 回     │
         │                     │
         │  [キャンセル] [実行] │
         └─────────────────────┘
```

- **▶ Run（左側）**: 1クリックで即実行。従来と完全に同じ動作
- **▼（右側）**: ドロップダウンを開き「繰り返し実行...」を選択
- **回数指定**: ポップオーバーで回数を入力し「実行」で開始。明示的な確認ステップにより誤操作を防止
- **実行中表示**: 「⏳ Running (3/10)...」のようにイテレーション進捗を表示
- **完了サマリー**: 「✅ 10/10 passed」または「⚠️ 8/10 passed, 2 failed」を表示
- **詳細結果**: 各イテレーションの結果は展開して確認可能

#### 5. Phase B への拡張パス

Phase A の実行ループがそのまま Phase B の基盤になる:

```
Phase A: for i in 1..N:  run(story, 固定値)
Phase B: for row in dataSource:  run(story, row の値で step.value を差し替え)
```

Phase B で追加するもの:
- データソース連携（CSV / Google Spreadsheet）
- Step の `value` にプレースホルダー構文（例: `{{商品名}}`）
- 実行時にプレースホルダーをデータ行の値で置換

Phase A の `RepeatResult` 型・進捗通知・中断機能はそのまま流用できる。

### 6. 実行エンジンの安定性（DOM 安定待機）

リピート実行で安定性検証を行うには、実行エンジン自体が安定していなければならない。
現状の `SLOW_MO = 300ms` 固定待機では、SPA のレンダリング速度次第で assert が古い DOM を見て失敗する。

#### 課題

- click / type / select 後、React 等のフレームワークが DOM を更新するまでのラグが不定
- 固定待機は遅い環境では足りず、速い環境では無駄に待つ
- リピート実行ではこの問題が累積し、「本来パスするシナリオが不安定に失敗する」

#### 解決策: MutationObserver ベースの DOM 安定待機

アクション実行後、DOM の変更が一定時間途絶えるまで待つ「settle」関数を導入する。

```javascript
function waitForDomSettle(timeoutMs, quietMs) {
  // MutationObserver で DOM 変更を監視
  // quietMs の間変更がなければ「安定した」と判断
  // timeoutMs を超えたら安定しなくても続行（無限待ちを防止）
}
```

- **quietMs**: DOM 変更が途絶えてから「安定」と見なすまでの静寂時間（デフォルト 150ms）
- **timeoutMs**: 最大待機時間（デフォルト 3000ms）。SPA が延々とアニメーションする場合のガード
- **適用タイミング**: click / type / select の完了後。navigate は既に readyState ポーリングがあるので不要
- **SLOW_MO 廃止**: 固定待機を DOM 安定待機に置き換え。ただし操作の視認性のため最小待機 50ms は残す

## Consequences (影響)

### メリット
- 既存の Story データモデルを変更しない（実行時パラメータのみ）
- データ量産・安定性検証のユースケースに対応できる
- Phase B（データドリブン）への拡張が自然にできる設計

### デメリット
- ブラウザ毎回起動のため、大量繰り返し時はそれなりに時間がかかる
- 結果データが繰り返し回数分に増えるため、結果表示 UI の設計が必要

### 影響を受ける既存 ADR

| ADR | 影響 |
|-----|------|
| ADR-008 | 拡張 — 実行結果に RepeatResult を追加、進捗通知の仕組みを追加 |
| ADR-005 | 関連 — Phase B でデータソース連携と合流する |
