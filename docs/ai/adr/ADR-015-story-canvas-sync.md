# ADR-015: Canvas の廃止と Story 中心アーキテクチャへの移行

- 日付: 2026-04-03
- ステータス: 承認

## Context (背景)

### 経緯

当初、Figma / Google Spreadsheet 連携を見据えて Canvas（ReactFlow ベースの画面遷移図）を中心にUIを設計した。Canvas 上でノード（画面）とエッジ（遷移）を配置し、エッジに Story を紐づける構造だった。

その後、REC（録画）ベースの Story 作成が主要なワークフローとなり、設計の前提が変わった。

### 現状の問題

1. **不自然なデータモデル**: UIから先に作った結果、Story = Canvas エッジ（ID共有）という構造になった。1エッジ = 2ノード間なので、3画面以上をまたぐ Story を表現できない
2. **Story の従属関係**: Story がエッジに従属しており、本来の主従関係（Story が主）が逆転している
3. **Canvas の存在意義の喪失**: REC ベースに移行した時点で「手動で画面遷移を設計する」ユースケースがなくなり、Canvas はコア体験に寄与していない
4. **複雑さの原因**: Canvas と Story の同期問題（スタンドアロン Story が表示されない等）が継続的に発生している

### あるべき姿

- Story は Playwright の1テストと同等の独立した存在（複数画面をまたぐ操作の列）
- REC で録画したら、1つの Story ができる（画面数に制約なし）
- コア体験は **REC + Story リスト + テスト実行**

## Decision (決定)

### Canvas を廃止し、Story 中心のアーキテクチャに移行する

1. **Canvas（ReactFlow）を UI から削除する**
2. **Story は完全に独立したエンティティ** — `{ id, title, baseUrl?, steps[] }` のデータモデルは現行のまま
3. **Story リストがメインのナビゲーション** — Canvas に代わる Story 管理UI
4. **コア体験を REC + Story リスト + テスト実行に集中する**

### やること

- Canvas コンポーネント（ReactFlow）と関連コード（ノード/エッジ管理）を削除
- Story と Edge の ID 共有を解消し、Story を独立管理に変更
- Canvas に代わる Story リストUIを実装
- nodes.json / edges.json の永続化を廃止

### やらないこと

- Figma / Spreadsheet 連携（当面スコープ外）
- 画面遷移の可視化（明確なユースケースが出てから再検討）

## Consequences (影響)

### ポジティブ

- Story が Playwright の1テストと同じ粒度になり、自然なモデルになる
- 3画面以上をまたぐ Story を制約なく作成できる
- Canvas-Story 同期問題が根本的に解消される
- コードベースが大幅にシンプルになる（ReactFlow 依存の削除）
- コア体験（REC → Story → テスト実行）に集中できる

### ネガティブ

- Canvas ベースの画面遷移設計UIが使えなくなる
- 将来、可視化が必要になった場合は再実装が必要

### 将来の拡張ポイント

- 画面カバレッジの可視化など、明確なユースケースが出た時点で Canvas を再導入できる
- その際は Story が主、可視化が従の関係で設計する
