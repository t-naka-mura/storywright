# ADR-001: ドキュメント駆動開発の採用

- 日付: 2026-03-31
- ステータス: 承認

## Context (背景)

Storywrightは構想段階にあり、UIも未確定。AIとの協業で開発を進めるため、意思決定の過程を記録し、仕様の全容を追跡可能にする仕組みが必要。

ChatGPTとの協業で一度コンテキスト（CLAUDE.mdなど）が失われた経験から、ドキュメントをコードベースに含めて管理する重要性が明確になった。

## Decision (決定)

以下の方針でドキュメント駆動開発を採用する。

1. **docs/ai/ をプロダクトドキュメントの中心に据える**
   - product-overview.md でプロダクトの全体像を管理
   - 機能仕様は docs/ai/ 配下に spec-*.md として管理
   - 設計判断は ADR として docs/ai/adr/ に記録

2. **実装よりドキュメントを先に書く**
   - 新機能は仕様ドキュメント → ADR（必要に応じて） → 実装の順で進める
   - AIとの協業時は、ドキュメントで方針を合意してから実装に入る

3. **CLAUDE.md をプロジェクトのエントリポイントとする**
   - AIが最初に読むファイルとして、プロジェクト概要と開発ルールを記載
   - docs/ai/ への導線を含める

## Consequences (影響)

### メリット
- AIとのコンテキスト共有が安定する（会話が途切れてもドキュメントが残る）
- 意思決定の経緯が追跡可能になる
- UI未確定の段階でも、仕様レベルで議論・合意を進められる

### デメリット
- ドキュメントの維持コストが発生する
- 過度なドキュメント化は開発速度を下げるリスクがある

### 軽減策
- ドキュメントは「必要十分」を心がけ、形式より内容を重視する
- 陳腐化したドキュメントは積極的に更新・削除する
