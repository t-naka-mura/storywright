# ADR-006: UI/UXアーキテクチャ

- 日付: 2026-03-31
- ステータス: 承認
- 更新: 2026-04-01（キャンバス + リストのハイブリッド方針を追記）

## Context (背景)

Storywright は非エンジニアも利用するデスクトップアプリである。UIは直感的であるほど良い。Figma/FigJam のようなキャンバスベースのUIが、ストーリー（画面遷移フロー）の可視化に適している。

一方で、Spreadsheet からのインポートやストーリーの一括管理など、リスト型の方が効率的な場面もある。E2Eテストツールの既存製品（mabl, Autify, Katalon 等）はほぼリスト/テーブル型であり、ノードベースのみでは編集効率に課題がある。

## Options (選択肢)

### A. キャンバスベース（React Flow）のみ

ノード（画面）とエッジ（ストーリー）でフローを表現。

**メリット**: 直感的、非エンジニアが理解しやすい、画面遷移が視覚化される
**デメリット**: ストーリーが増えるとキャンバスが混雑、一覧性・編集効率が低い、Spreadsheet インポートとの相性が悪い

### B. リスト/テーブル型のみ

ストーリー一覧をテーブルで表示、クリックで詳細編集。

**メリット**: 実装が容易、一覧性が高い、Spreadsheet との相性が良い
**デメリット**: 画面遷移フローが見えにくい、非エンジニアには馴染みにくい

### C. カンバン型（Trello風）

ステータス別にストーリーをカード表示。

**メリット**: 進捗管理に向く
**デメリット**: フロー表現ができない、テスト実行との相性が悪い

### D. キャンバス + リストのハイブリッド

キャンバスで俯瞰・可視化、リストで一覧管理・編集。ビュー切り替えで併用。

**メリット**: それぞれの強みを活かせる、Spreadsheet 連携とフロー可視化を両立
**デメリット**: 2つのビューの同期・整合性維持が必要、実装コストが高い

## Decision (決定)

**D. キャンバス + リストのハイブリッドを採用する。**

当初は A（キャンバスのみ）で決定していたが、以下の検討を経てハイブリッド方針に変更した。

- E2Eテストツールでノードベースを採用した既存製品はほぼ存在しない
- キャンバスの価値は「テスト作成」よりも「テストカバレッジの可視化」にある
- Spreadsheet インポートはリスト型との相性が良い
- 非エンジニアの「全体像把握」と「詳細編集」は別のUIが最適

### 各ビューの役割

| ビュー | 役割 | 主なユースケース |
|--------|------|----------------|
| **キャンバス** | 俯瞰・可視化 | テストカバレッジの把握、画面遷移の全体像、テスト実行結果の概観 |
| **リスト** | 一覧・編集 | Spreadsheet インポート結果の確認・編集、ストーリーの一括管理、検索・フィルタ |

### ユーザーフロー

```
1. Spreadsheet からインポート → リストビューで確認・編集
2. 画面にマッピング → キャンバスに配置
3. テスト実行 → キャンバスで結果俯瞰、失敗箇所はリストで詳細確認
```

### レイアウト構成

```
┌──────────────────────────────────────────────────┐
│ Toolbar                                           │
│ [+ 画面追加] [+ ストーリー追加] [Import] [▶ Run All]│
│                            [Canvas | List] toggle │
├──────────────────────────────────────────────────┤
│                    │                              │
│  Canvas View       │  Detail Panel                │
│  ┌─────────────┐   │  (共通: 選択に応じて切替)      │
│  │ [画面A]──→   │   │                              │
│  │    [画面B]   │   │  ストーリー名: ログインフロー    │
│  └─────────────┘   │  1. navigate: /login          │
│                    │  2. type: #email → test@...   │
│  ── or ──          │  3. click: .submit            │
│                    │  4. assert: .title → "..."    │
│  List View         │                              │
│  ┌─────────────┐   │  [▶ Run] [Edit] [Delete]      │
│  │ ☐ ログイン   │   │                              │
│  │ ☐ 新規登録   │   │                              │
│  │ ☐ 商品購入   │   │                              │
│  └─────────────┘   │                              │
├──────────────────────────────────────────────────┤
│ Status Bar                                        │
└──────────────────────────────────────────────────┘
```

- **Detail Panel は共通**: どちらのビューでもストーリー選択時に同じパネルを表示
- **ビュー切り替え**: Toolbar 上のトグルで Canvas / List を切り替え
- **データは共通**: 同一のストーリーデータを2つのビューが参照

### UI要素とデータモデルの対応

| UI要素 | データ | 操作 |
|--------|--------|------|
| ノード（Canvas） | 画面（URL + screenshot） | ドラッグで配置、ダブルクリックで編集 |
| エッジ（Canvas） | ストーリー | クリックでステップ編集パネル表示 |
| 行（List） | ストーリー | クリックでステップ編集パネル表示 |
| ステップ編集パネル | Story.steps[] | action/target/value の編集（共通） |
| ツールバー | — | 画面追加、インポート、テスト一括実行、ビュー切り替え |

### 技術選定

| ライブラリ | 用途 |
|-----------|------|
| React Flow | キャンバス（ノード、エッジ、ズーム、パン） |
| （CSSフレームワーク未定） | ツールバー、パネル、フォーム、リストビュー |

## Consequences (影響)

### メリット
- 画面遷移フローが視覚化され、テストカバレッジの把握が容易
- リストビューにより Spreadsheet 連携やストーリーの一括管理が効率的
- Detail Panel を共通化することで、どちらのビューでも一貫した編集体験
- 非エンジニアは好みのビューを選べる

### デメリット
- 2つのビューの実装・メンテナンスコスト
- キャンバスとリストのデータ同期の整合性管理
- React Flow の学習コスト

### 実装の優先順位

1. **Phase 1**: キャンバスビュー（現在実装済み）
2. **Phase 2**: リストビュー + ビュー切り替えトグル
3. **Phase 3**: テスト実行結果のキャンバス上での表示方法（エッジの色変え等）

### 今後の検討事項
- CSSフレームワーク / コンポーネントライブラリの選定
- リストビューの検索・フィルタ・ソート仕様
