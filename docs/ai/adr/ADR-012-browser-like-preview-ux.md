# ADR-012: ブラウザライクな Preview UX

- 日付: 2026-04-03
- ステータス: 提案

## Context (背景)

### 現状の課題

Preview パネルは単一の webview を表示するだけで、ブラウザとしての基本操作が欠けている:

1. **ナビゲーション不可**: 戻る/進むボタンがない。リンクで遷移した後、前のページに戻れない
2. **URL バーが連動しない**: ページ遷移しても Toolbar の Base URL が更新されない。URL を変更してもリロードされない
3. **タブがない**: 別タブが開くリンク（`target="_blank"` など）をクリックしても追跡できない。複数ページを並行して操作できない
4. **リロードできない**: ページを手動でリロードする手段がない

### あるべき体験

録画対象のサイトを操作する Preview は、Chrome のタブを使う感覚に近づけたい。ユーザーが「ブラウザとして当たり前にできること」ができないと、録画シナリオの幅が狭まる。

## Options (選択肢)

### A. Toolbar にナビゲーションボタンを追加（最小構成）

- 戻る・進む・リロードボタンを Toolbar に追加
- URL 入力を Enter で確定 → webview をナビゲート
- webview の遷移に URL 表示を追従させる
- タブは非対応

**メリット**: 実装が軽い、既存レイアウトに収まる
**デメリット**: 複数タブが使えない

### B. ブラウザバー + タブバーを Preview 領域に追加

- Preview 領域の上部にブラウザバー（← → ↻ + URL 表示）とタブバーを配置
- Toolbar の Base URL 入力は「初期 URL」の設定として残す
- 各タブが独立した webview を持つ
- タブの追加・閉じる・切り替えが可能
- `target="_blank"` で自動的に新タブが開く

**メリット**: Chrome に近い操作感、複数タブ対応
**デメリット**: UI の複雑さが増す、webview の管理が煩雑

### C. B + Toolbar の URL 入力をブラウザバーに統合

- Toolbar の Base URL 入力を廃止し、ブラウザバーの URL 欄に統合
- ブラウザバーが「アドレスバー」として機能（Enter で遷移、履歴ドロップダウン）
- Toolbar はアプリ機能（REC, Canvas/Preview 切替, Panel）に専念

**メリット**: Chrome の操作モデルに最も近い、URL 入力が1箇所に集約
**デメリット**: Toolbar のレイアウト変更が必要

## Decision (決定)

**C. ブラウザバー + タブバーを Preview に追加し、Toolbar の URL 入力を統合する。**

理由:
- URL 入力が Toolbar とブラウザバーの2箇所にあると混乱する
- 「Chrome のアドレスバー」は全ユーザーが知っているメンタルモデル
- Toolbar を整理することで、録画操作（REC / Assert）がより目立つ

### 変更内容

#### 1. Preview 領域のレイアウト

```
┌─ Preview ──────────────────────────────────────────┐
│ Tab Bar                                             │
│ ┌──────┐ ┌──────┐ ┌───┐                            │
│ │ Tab 1│ │ Tab 2│ │ + │                            │
│ └──────┘ └──────┘ └───┘                            │
│ Browser Bar                                         │
│ ← → ↻ │ https://example.com/login           │ ▼ │ │
│                                                     │
│ ┌─ webview ───────────────────────────────────────┐ │
│ │                                                 │ │
│ │              (ページ内容)                        │ │
│ │                                                 │ │
│ └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

#### 2. タブバー

- 各タブにタイトル（ページの `<title>`）と ✕ ボタン
- `+` ボタンで空白タブを追加
- `target="_blank"` リンクで自動的に新タブが開く
- タブ切り替えで webview を切り替え
- 最後のタブを閉じたら空白タブを表示

#### 3. ブラウザバー

- **← 戻る**: `webview.goBack()`。`canGoBack()` が false なら disabled
- **→ 進む**: `webview.goForward()`。`canGoForward()` が false なら disabled
- **↻ リロード**: `webview.reload()`
- **URL 欄**: 現在の URL を表示。編集して Enter で遷移（`webview.loadURL()`）
- **▼ 履歴**: クリックで URL 履歴ドロップダウン表示（既存の useUrlHistory を流用）

#### 4. URL の流れ

```
① ブラウザバーに URL 入力 → Enter
② webview.loadURL(url)
③ webview の did-navigate → ブラウザバーの URL 表示を更新
④ did-finish-load → URL 履歴に自動保存
```

- ブラウザバーの URL は webview の実際の URL に追従する（Chrome と同じ）
- ページ内のリンクで遷移しても URL 表示が更新される
- 戻る/進むでも URL が追従する

#### 5. Toolbar の変更

- Base URL 入力欄を削除
- Toolbar はアプリ操作に専念: `[Logo] [Preview | Canvas] [+ 画面追加] [Import] ... [REC/Stop] [Assert] [Panel] [Run All]`
- Story の `baseUrl` はブラウザバーの初期 URL として使う

#### 6. 録画との連携

- 録画中（REC）は全タブのイベントをキャプチャする
- タブ切り替え時、新タブの webContents にもレコーダーをアタッチする
- タブ間の遷移は `navigate` ステップとして記録される
- `allowpopups` + `new-window` イベントで新タブのwebContents を追跡

#### 7. 実装フェーズ

**Phase 1（最小限のブラウザ体験）**:
- ブラウザバー追加（← → ↻ + URL 表示 + Enter で遷移）
- URL 追従（webview の遷移に連動）
- URL 履歴ドロップダウン
- Toolbar の URL 入力を削除

**Phase 2（タブ対応）**:
- タブバー追加（追加・閉じる・切り替え）
- `target="_blank"` で自動新タブ
- 各タブに独立した webview
- 録画時の複数タブ追跡

## Consequences (影響)

### メリット
- Chrome と同じ操作感 — ユーザーが学習コストゼロで使える
- URL 入力が1箇所に集約 — 混乱が減る
- 複数タブ対応で、別タブが開く決済フローなどの録画が可能になる
- 戻る/進むで試行錯誤しながら録画できる

### デメリット
- Preview 領域に UI 要素が増え、表示領域がやや狭くなる
- 複数 webview の管理でメモリ使用量が増加
- 録画ロジックが複数タブ分に複雑化

### 影響を受ける既存 ADR

| ADR | 影響 |
|-----|------|
| ADR-009 | 拡張 — Preview のレイアウト変更、録画の複数タブ対応 |
| ADR-010 | 軽微 — 録画フローの起点がブラウザバーに変わる |
| ADR-006 | 変更 — Toolbar から URL 入力を削除、Preview 領域のレイアウト変更 |
