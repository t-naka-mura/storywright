# ADR-002: E2Eテストフレームワークの選定

- 日付: 2026-03-31
- ステータス: 承認

## Context (背景)

Storywrightはユーザーストーリーに基づいてE2Eテストを実行するアプリである。コアとなるE2Eテストフレームワークの選定が必要。

Tauriデスクトップアプリから外部のWebサイト/アプリに対してE2Eテストを実行する想定のため、ブラウザ自動化の能力が必須。また、非エンジニアが作成したユーザーストーリーからテストを生成・実行する仕組みを考慮すると、プログラムからの制御しやすさも重要。

## Options (選択肢)

### A. Playwright

| 観点 | 評価 |
|------|------|
| ブラウザ対応 | Chromium, Firefox, WebKit |
| 言語 | TypeScript/JS, Python, Java, C# |
| プログラムからの制御 | API が豊富、ヘッドレス/ヘッドフル切替可 |
| 並列実行 | ネイティブサポート |
| コミュニティ | 活発（Microsoft） |
| Tauri との相性 | Node.js ライブラリとして呼び出し可能 |

**メリット**: マルチブラウザ、高速、APIが豊富でプログラムからの制御に向く
**デメリット**: バンドルサイズが大きい（ブラウザバイナリ含む）

### B. Cypress

| 観点 | 評価 |
|------|------|
| ブラウザ対応 | Chromium系, Firefox（WebKit実験的） |
| 言語 | JavaScript/TypeScript のみ |
| プログラムからの制御 | モジュールAPI あるが Playwright ほど柔軟ではない |
| 並列実行 | 有料プラン（Cypress Cloud）が必要 |
| コミュニティ | 活発 |
| Tauri との相性 | Node.js から利用可能 |

**メリット**: デバッグUI が優秀、学習コストが低い
**デメリット**: マルチタブ非対応、並列実行に制約、プログラム制御の柔軟性が劣る

### C. Puppeteer

| 観点 | 評価 |
|------|------|
| ブラウザ対応 | Chromium のみ（Firefox実験的） |
| 言語 | JavaScript/TypeScript のみ |
| プログラムからの制御 | 低レベルAPI、柔軟 |
| 並列実行 | 自前で実装が必要 |
| コミュニティ | 活発（Google） |
| Tauri との相性 | Node.js から利用可能 |

**メリット**: 軽量、低レベル制御が可能
**デメリット**: Chromium限定、テストランナー機能がなく自前構築が多い

## Decision (決定)

**Playwright を採用する。ブラウザは同梱せず、システムブラウザを利用する。**

- マルチブラウザ対応が必要 → Playwright が最適
- テスト実行UIはStorywright独自で構築する → Playwright のAPIは制御性が高く適している
- ブラウザは同梱しない → サイズ肥大化とバージョン追従の困難さを回避
- システムブラウザを使用 → Playwright の `channel` オプションで Chrome/Edge/Firefox を制御
- 非エンジニアも利用するため、セットアップの簡易さを重視

## Consequences (影響)

### メリット
- Chromium, Firefox, WebKit のマルチブラウザテストが可能
- 豊富なAPIでストーリーからのテスト生成・実行をプログラム的に制御できる
- ブラウザ非同梱によりアプリサイズを抑制できる
- ユーザーが実際に使っているブラウザでテストできる（E2Eとしてより実践的）

### デメリット
- ユーザーのPCにブラウザがインストールされている必要がある（通常は問題にならない）
- ブラウザバージョンによる挙動差が出る可能性がある
- Playwright のバージョンアップに追従する必要がある
