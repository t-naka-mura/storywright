# ADR-006: UI/UXアーキテクチャ

- 日付: 2026-03-31
- ステータス: 承認

## Context (背景)

Storywright は非エンジニアも利用するデスクトップアプリである。UIは直感的であるほど良い。Figma/FigJam のようなキャンバスベースのUIが、ストーリー（画面遷移フロー）の可視化に適している。

## Options (選択肢)

### A. キャンバスベース（React Flow）

ノード（画面）とエッジ（ストーリー）でフローを表現。

**メリット**: 直感的、非エンジニアが理解しやすい、画面遷移が視覚化される
**デメリット**: 実装コストがリスト型より高い

### B. リスト/テーブル型

ストーリー一覧をテーブルで表示、クリックで詳細編集。

**メリット**: 実装が容易
**デメリット**: 画面遷移フローが見えにくい、非エンジニアには馴染みにくい

### C. カンバン型（Trello風）

ステータス別にストーリーをカード表示。

**メリット**: 進捗管理に向く
**デメリット**: フロー表現ができない、テスト実行との相性が悪い

## Decision (決定)

**A. React Flow によるキャンバスベースUIを採用する。**

### レイアウト構成

```
┌──────────────────────────────────────────────────┐
│ Toolbar                                           │
│ [+ 画面追加] [+ ストーリー追加] [Import] [▶ Run All]│
├──────────────────────────────────────────────────┤
│                    │                              │
│                    │  Step Editor / Detail Panel   │
│   Canvas           │                              │
│   (React Flow)     │  ストーリー名: ログインフロー    │
│                    │  1. navigate: /login          │
│   [画面A]──→[画面B] │  2. type: #email → test@...   │
│                    │  3. click: .submit            │
│                    │  4. assert: .title → "..."    │
│                    │                              │
│                    │  [▶ Run] [Edit] [Delete]      │
└──────────────────────────────────────────────────┘
```

### UI要素とデータモデルの対応

| UI要素 | データ | 操作 |
|--------|--------|------|
| ノード | 画面（URL + screenshot） | ドラッグで配置、ダブルクリックで編集 |
| エッジ | ストーリー | クリックでステップ編集パネル表示 |
| ステップ編集パネル | Story.steps[] | action/target/value の編集 |
| ツールバー | — | 画面追加、インポート、テスト一括実行 |

### 技術選定

| ライブラリ | 用途 |
|-----------|------|
| React Flow | キャンバス（ノード、エッジ、ズーム、パン） |
| （CSSフレームワーク未定） | ツールバー、パネル、フォーム |

## Consequences (影響)

### メリット
- 画面遷移フローが視覚化され、非エンジニアでも直感的に理解・操作できる
- React Flow が標準でドラッグ＆ドロップ、ズーム、パンを提供
- React エコシステム上で自然に統合できる

### デメリット
- React Flow の学習コスト
- キャンバスUIの操作性チューニングが必要

### 今後の検討事項
- CSSフレームワーク / コンポーネントライブラリの選定
- テスト実行結果のキャンバス上での表示方法（エッジの色変え等）
